SCORING METHODOLOGY

How Scores Are Measured

NiCE Designer calculates separate compliance scores for SOL tokens, Lyra tokens, and accessibility. Here's exactly how each number is computed and what it means.

Overview

Four Independent Scores

NiCE Designer runs four independent audits and produces four scores. Each one is a percentage from 0 to 100 and they are entirely independent of each other. A frame can be 100% SOL compliant and 0% Lyra compliant at the same time -- that is expected and correct behavior.

◈
SOL Compliance
How well the selection uses SOL design tokens across all five categories.
◈
Lyra Compliance
How well the selection uses Lyra design tokens across all five categories.
◈
Min Accessibility
Passes minimum WCAG AA requirements: contrast, sizing, and focus visibility.
◈
Full Accessibility
Fully meets WCAG AA across all criteria including ARIA, headings, and tap targets.

The Formula

How the Number Is Calculated

Every audit category uses the same formula. The tool inspects every relevant property in the selection, counts how many have an issue, and divides the clean count by the total.

Score = ((total − issues) / total) × 100
Example

"15/147 Colors" means 15 color properties have issues out of 147 total inspected.
Score = ((147 − 15) / 147) × 100 = 89.8%, shown as 90%.

Since v9 each category uses its own denominator, not a global total. A file with many more spacing properties than color properties will not have its color score affected by that imbalance.

Audit Categories

The Five Compliance Categories

SOL and Lyra compliance are each measured across five dimensions. Each dimension has its own score that rolls up into the overall compliance percentage.

Colors
v10: semantic roles
What's checked
Every fill, stroke, and background color on every node in the selection.
What counts as an issue
A raw hex value where a design token should be used; a token from the wrong system (e.g. a Lyra token present in SOL mode); a token that semantically mismatches its role (a text color token applied as a background).
Semantic matching (v10)
The token lookup considers the role of the property (text, background, border, icon) to flag semantic mismatches more accurately and suggest correct alternatives.
Text Styles
v9: family-aware
What's checked
Font family, font size, font weight, and line height for every text node in the selection.
What counts as an issue
A font family not in the active design system; a font size or weight not matching any token; a wrong-system font (Inter flagged in SOL mode gets a role-equivalent suggestion).
Family-aware matching (v9)
Family, size, and weight are all considered together. A family not in the current system at all is flagged as wrong-system rather than just out-of-range.
Spacing
auto-binding detection
What's checked
Padding, gap, and margin-equivalent properties. In Figma: horizontal padding, vertical padding, item spacing, and counter-axis spacing.
What counts as an issue
A numeric value that doesn't match any spacing token; an unbound variable where the value matches a token (a missed binding still counts as an issue).
Auto-binding detection
The audit identifies values that match a known token but aren't bound to the variable. This surfaces easy wins for the conversion tool.
Radius
v9: corner binding fix
What's checked
Corner radius properties on frames and components -- all four corners individually plus the unified radius value.
What counts as an issue
A raw numeric radius not matching any radius token; a unified radius not bound to a variable (Figma requires binding all four individual corners, not just cornerRadius itself).
v9 fix
Radius variable binding was corrected in v9 to bind all four corner properties individually, matching Figma's variable resolution behavior.
Components
What's checked
Whether component instances on the canvas are sourced from the active design system library (SOL or Lyra).
What counts as an issue
A detached component (no longer linked to any library); a component from the wrong library; or a component that doesn't exist at all in the SOL or Lyra system.

Accessibility

The Two Accessibility Scores

Accessibility is measured separately from token compliance and uses the number of audited nodes as the denominator -- so scores are comparable across frames of different sizes.

Min Accessibility
WCAG AA baseline
What's covered
Color contrast ratios (WCAG AA: 4.5:1 for normal text, 3:1 for large text and UI components); text sizing minimums; interactive element focus visibility.
To score 100%
A frame must pass every min-a11y check with zero issues. Any single contrast failure or missing focus indicator brings the score below 100%.
Full Accessibility
WCAG AA complete
What's covered
Everything in Min Accessibility, plus: missing alt text on images; missing ARIA labels on interactive elements; heading hierarchy; keyboard navigation order; touch target sizing (44x44px minimum).
To score 100%
A frame must have zero issues across all full-a11y criteria. The score divides issue count by the total nodes audited.

Fixes

What Counts as a Fix

A fix is any change that moves a property from an issue state to a compliant state. The plugin logs each fix in the edit history (since v10) and allows undo per step.

  • ✓
    Binding a raw color value to the correct design token in the active system.
  • ✓
    Swapping a wrong-system token for the correct-system equivalent (e.g. a SOL color token replaced with its Lyra counterpart).
  • ✓
    Binding an unbound spacing or radius value to the matching token variable.
  • ✓
    Converting a detached component back to a live library instance from the active system.
  • ✓
    Adding a missing ARIA label, alt text attribute, or fixing a color contrast ratio below threshold.

Snoozing

Snoozed Items and Score Calculation

Since v10, Apply All automatically snoozes fixed items so they disappear from the issue list. Snoozed items are excluded from the score calculation entirely, so the displayed score always reflects only the remaining open issues.

Since v10 Auto-snooze behavior

When a fix is applied via the plugin, the affected issue is snoozed automatically. The denominator in the score formula adjusts to exclude snoozed properties, so your score always reflects what's actually still open. You can toggle auto-snooze in plugin settings, and manually un-snooze any item to bring it back into scope.

Tips

Getting the Most from Scores

1

Run the audit before and after conversion to verify your score improved. The before/after delta is the clearest signal that the conversion worked correctly.

2

A score of 75%+ means most issues are minor. Focus on Components and Colors first -- they have the biggest visual impact and are the most visible in design reviews.

3

Use the manual-fixes-checklist skill after running automated fixes to see what the automation couldn't handle and what still needs a human decision.
